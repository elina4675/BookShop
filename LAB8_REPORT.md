# Лабораторна робота №8

## Тема
Безперервна інтеграція та тестування (Continuous Integration and Testing — CI/CD)

## Мета роботи
Ознайомитися з концепцією CI/CD, зрозуміти роль автоматизованого тестування в процесі безперервної інтеграції та набути практичних навичок налаштування базового CI-пайплайну за допомогою GitHub Actions.

## Короткий опис проєкту та інструментів
Для виконання лабораторної роботи використано проєкт `BookShop`, який містить PHP-файли `index.php` і `config.php`.
Для автоматизованого тестування додано окремий набір автотестів на Python з використанням стандартного модуля `unittest`.
Автотести перевіряють:

1. коректність основних параметрів конфігурації в `config.php`;
2. маршрутні значення за замовчуванням у `index.php`;
3. шаблон побудови шляху до контролера та підключення головного шаблону.

Для безперервної інтеграції використано `GitHub Actions`.

## Локальний запуск автотестів
```bash
python3 -m unittest discover -s tests -p "test_*.py" -v
```

## Лістинг конфігураційного файлу CI/CD
Файл: `.github/workflows/test-pipeline.yml`

```yaml
name: Automated Tests CI

on:
  push:
    branches: ["main"]
  pull_request:
    branches: ["main"]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.13"

      - name: Run Tests
        run: python -m unittest discover -s tests -p "test_*.py" -v
```

## Негативний сценарій

```python
self.assertEqual(constants["NUMBER_OF_ROWS_IN_PAGE"], 10)
```

## Висновки
Під час виконання лабораторної роботи було налаштовано базовий CI-пайплайн для проєкту `BookShop` за допомогою `GitHub Actions`.
До репозиторію було додано автотести, які автоматично запускаються при `push` та `pull_request` у гілку `main`.
У ході роботи підтверджено, що CI дозволяє автоматично виявляти помилки в коді: при коректному коді пайплайн завершується успішно, а при навмисно зламаному тесті — завершується з помилкою та показує детальну причину збою в логах.
Отже, використання CI/CD підвищує якість програмного забезпечення, прискорює виявлення дефектів і зменшує ризик потрапляння помилкового коду до основної гілки проєкту.

## Відповіді на контрольні запитання
1. **У чому головна відмінність між Continuous Delivery та Continuous Deployment?**  
   `Continuous Delivery` автоматично готує застосунок до релізу, але розгортання на production виконується вручну. `Continuous Deployment` після успішного проходження всіх етапів автоматично розгортає зміни без втручання людини.

2. **Що станеться, якщо під час виконання CI-пайплайну хоча б один автотест завершиться невдало (Failed)?**  
   Workflow буде позначений як `failed`, відповідний job завершиться помилкою, а в логах буде показано, який саме тест не пройшов і з якої причини. Такий код не повинен зливатися в основну гілку без виправлення.

3. **Яку функцію виконують "Runners" у GitHub Actions?**  
   `Runners` — це віртуальні машини або середовища виконання, на яких GitHub Actions запускає workflow, jobs і окремі steps.

4. **Навіщо у конфігураційному файлі пайплайну потрібен крок checkout code?**  
   Крок `checkout code` завантажує файли репозиторію на runner, щоб наступні кроки могли працювати з кодом, запускати збірку та тести.

5. **Назвіть щонайменше три події (Events), які можуть слугувати тригером для запуску автоматизованих тестів у репозиторії.**  
   Прикладами таких подій є `push`, `pull_request`, `workflow_dispatch`, `schedule`, `release`.
